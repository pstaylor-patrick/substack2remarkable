1. Cloudflares latest outage proves dangers of global configuration changes (again)

A mere two weeks after Cloudflare suffered a major outage and took down half the internet, the same thing has happened again. Last Friday, 5th December, thousands of sites went down or partially down once more, in a global Cloudflare outage lasting 25 minutes.

As per last time, Cloudflare was speedy to share a full postmortem on the same day. It estimated that 28% of Cloudflares HTTP traffic was impacted. The cause of this latest outage was Cloudflare making a seemingly innocent but global configuration change that went on to take out a good portion of Cloudflare, globally, until being reverted. Heres what happened:

  • Cloudflare was rolling out a fix for a nasty React security vulnerability

  • The fix caused an error in an internal testing tool

  • The Cloudflare team disabled the testing tool with a global killswitch

  • As this global configuration change was made, the killswitch unexpectedly caused a bug that resulted in HTTP 500 errors across Cloudflares network

In this latest outage, Cloudflare was burnt by yet another global configuration change. The previous outage in November happened thanks to a global database permissions change. In the postmortem of that incident, the Cloudflare team closed with this action item:

Hardening ingestion of Cloudflare-generated configuration files in the same way we would for user-generated input

This change would make it so that Cloudflares configuration files do not propagate immediately to the full network, as they still do now. But making all global configuration files have staged rollouts is a large implementation that could take months. Evidently, there wasnt time to make it yet, and it has come back to bite Cloudflare.

Unfortunately for Cloudflare, customers are likely to find unacceptable a second outage with similar causes to a previous one, only weeks ago. If Cloudflare proves unreliable, customers should plan to onboard to backup CDNs at the very least, and a backup CDN vendor will do its best to convince new customers to use it as the primary CDN.

Cloudflares value-add rests on rock-solid reliability without customers needing to budget for a backup CDN. Yes, publishing postmortems on the same day as an outage occurs helps restore trust, but that will crumble anyway with repeated large outages.

To be fair, the company is doubling down on implementing staged configuration rollouts. In its postmortem, Cloudflare is its own biggest critic. CTO Dane Knecht reflected:

[Global configuration changes rolling out globally] remains our first priority across the organization. In particular, the projects outlined below should help contain the impact of these kinds of changes:

  • Enhanced Rollouts & Versioning: Similar to how we slowly deploy software with strict health validation, data used for rapid threat response and general configuration needs to have the same safety and blast mitigation features. This includes health validation and quick rollback capabilities among other things.

  • Streamlined break glass capabilities: Ensure that critical operations can still be achieved in the face of additional types of failures. This applies to internal services as well as all standard methods of interaction with the Cloudflare control plane used by all Cloudflare customers.

  • Fail-Open Error Handling: As part of the resilience effort, we are replacing the incorrectly applied hard-fail logic across all critical Cloudflare data-plane components. If a configuration file is corrupt or out-of-range (e.g., exceeding feature caps), the system will log the error and default to a known-good state or pass traffic without scoring, rather than dropping requests. Some services will likely give the customer the option to fail open or closed in certain scenarios. This will include drift-prevention capabilities to ensure this is enforced continuously.

These kinds of incidents, and how closely they are clustered together, are not acceptable for a network like ours.

Global configuration errors often trigger large outages

Theres a pattern of implicit or explicit global configuration errors causing large outages, and some of the biggest ones in recent years were caused by a single change being rolled out to a whole network of machines:

  • DNS and DNS-related systems like BGP: DNS changes are global by default, so its no wonder that DNS changes can cause global outages. Metas 7-hour outage in 2021 was related to DNS changes (more specifically, Border Gateway Protocol changes.) Meanwhile, the AWS outage in October started with the internal DNS system.

  • OS updates happening at the same time, globally: Datadogs 2023 outage cost the company $5M and was caused by Datadogs Ubuntu machines executing an OS update within the same time window, globally. It caused issues with networking, and it didnt help that Datadog ran its infra on 3 different cloud providers across 3 networks. The same kind of Ubuntu update also caused a global outage for Heroku in 2024.

Globally replicating configs: in 2024, a configuration policy change was rolled out globally and crashed every Spanner database node straight away. As Google concluded in its postmortem: Given the global nature of quota management, this metadata was replicated globally within seconds.

Implementing gradual rollouts for all configuration files is a lot of work. Its also invisible labor because when done well, then its benefits will be undetectable, except in the absence of incidents, thanks to better infrastructure!

The largest systems in the world will likely have to implement safer ways to roll out configs but not everybody needs to. Staged configuration rollout doesnt make much sense for smaller companies and products because this infra work slows down product development.

It doesnt just slow down building, but every deployment, too, and this friction is designed to make everything slower. As such, they dont make much sense unless the stability of mature systems is more important than fast iteration.

Software engineering is a field where tradeoffs are a fact of life, and universal solutions dont exist. The development which worked for a system with 1/100th of the load and users a year ago, may not make sense today.

2. Industry Pulse

Poor capacity planning at AWS?

Business Insider reports:

Over the summer, AWS Bedrock hit critical capacity constraints that drove some customers to rival services, such as Googles cloud service, according to an internal July document obtained by Business Insider.

The shortages led to tens of millions of dollars in lost or delayed revenue. Epic Games shifted a $10 million Fortnite project to Google Cloud after AWS failed to provide enough quota for Bedrock, according to the document.

Other customers, including Atlassian and GovTech Singapore, were waiting on quota increases this summer, delaying at least $52.6 million in projected sales, the document also disclosed.

If any company is good at estimating future demand and ensuring it has sufficient servers to serve it, that company is AWS. And yet, it seems AWS simply did not have enough GPUs to run all the models on its LLM platform (Bedrock) that customers wanted to use. As a result, they had to prioritize customers and play the game of try-not-to-lose-unhappy-customers.

If AWS had trouble coping with unexpected demand for AI usage, then this is probably also the case for all other cloud providers. It shows how much tech companies are investing into building and deploying LLMs, and operating products that make extensive use of them.

The big winners look like being Google Cloud and Anthropic (running on AWS and GCP). Also from Business Insider:

Customers using Anthropics Claude models through Bedrock opted to switch to Anthropics own platform or Google Cloud because of ongoing capacity, latency, and feature parity issues, according to the July AWS document. Companies such as Figma, Intercom, and Wealthsimple were among those migrating their workloads due to one or several of these challenges.

Thomson Reuters also chose Google Cloud over Bedrock for its CoCounsel AI product after finding AWSs service was 15% to 30% slower and lacked key government compliance certifications, the document showed.

Meta moving to closed AI approach

Bloomberg reports:

Metas next-generation model called] Avocado, is expected to debut sometime next spring, and may be launched as a closed model one that can be tightly controlled and that Meta can sell access to, according to people familiar with the matter, who declined to speak publicly about internal plans. The move, which aligns with what rivals Google and OpenAI do with their models, would mark the biggest departure to date from the open-source strategy Meta has touted for years.

Until now, Meta released open weight models its Llama family which are free to use for all, except Metas main competitors. My sense was that Meta was trying to throw a wrench into the business models of OpenAI and Anthropic by offering a free model which made it harder for rivals to charge for access to their closed models.

While I can see why an engineering team would be interested in self-hosting a Meta open source model, I dont understand why anyone would pay to access its next-generation AI model even if its as capable as OpenAIs or Anthropics.

You know what to expect from those AI companies, whereas Meta seems to keep changing its mind about its developer-facing product strategy. Also, it has been notoriously unreliable in selling products to developers and companies:

  • Shut down mobile-backend-as-a-service Parse in 2017, 4 years after acquiring it.

  • Shut down corporate social network Workplace for paying customers earlier this year, 10 years after launching it.

One reason an open model from Meta was so attractive is that the company cannot shut it down, so its safe to build on top of it. With any other Meta product, its hard to be confident that itll be supported unless it generates enough ad revenue which still accounts for ~97% of all income at the social media giant.

Do we face a RAM shortage?

Author and software developer Jeff Geerling warns of an upcoming RAM shortage, which data seems to back up:

The cause is manufacturers shifting RAM production to AI use cases, says Jeff:

The reason for all this, of course, is AI datacenter buildouts. I have no clue if theres any price fixing going on like there was a few decades agothats something conspiracy theorists can debatebut the problem is theres only a few companies producing all the worlds memory supplies.

And those companies all realized they can make billions more dollars making RAM just for AI datacenter products, and neglect the rest of the market.

So theyre shutting down their consumer memory lines, and devoting all production to AI.

In an eye-catching story by PC World, RAM prices are getting so high that Samsung Semiconductor reportedly refused a RAM order for new Galaxy phones made by Samsung Electronics because it was prioritizing data center customers who want to buy RAM too.

The best time to buy a RAM upgrade or a new machine might be now, before prices increase even more.

Timeframes to $600K total comp at Amazon and Meta

It takes around twice as long to earn a standout, $600K/year total compensation package at Amazon as it does at Meta, according to new research by Software compensation site Levels.fyi, which analyzed aggregated salary data from the tech giants.

Amazon and Meta are roughly equally difficult to land a job in, they each demand similarly high performance, and both brand names are very high profile. Needless to say, each pays at the top of the market, and its interesting to observe these differences. We covered more on tiers of compensation in Trimodal nature of tech compensation in the US, UK and India.

Apple bleeds execs to Meta

Heres something you dont see often: dozens of Apple executives quitting to join Meta, according to a chart, below. This finding is unusual because Apple has historically been extremely good at retaining executives, compared to other Big Tech companies. This is despite Meta and Google paying more; Apple tends to offer more stability, no mass layoffs, and the chance to work on cutting-edge, market-leading hardware like the iPhone.

Executives are usually very careful about changing jobs, as moving companies is a high-risk proposition for them. For an executive to thrive, the whole organization needs to perform well. If not, that executive could be pushed out, and finding the next job could take more than a year in the current conditions.

The fact some Apple execs are heading to Meta may suggest a few things:

  • Meta is offering eye-watering packages; possible double or more than peoples existing packages

  • Meta is serious about developing hardware, which should attract Apple execs

  • Apple is struggling to develop next-gen AI hardware devices, and struggling with AI in general. Meta is seen as more likely to build market-leading AI hardware and solutions

  • Trust in Apples top-level leadership may be weakening. Execs often quit when they lose faith in the C-level, or in the companys overall direction.

Coding-related income now the majority of Anthropics revenue?

An interesting stat shared by The Information is that Anthropic told investment banks that the majority of its revenue is tied to coding-related tasks. Eighty percent of Anthropics revenue comes from selling API access to Claude and it sounds like the majority of the API usage is to do with coding.

Anthropic really seems to have nailed its strategy for coding use cases and developers; no wonder Anthropic and Claude Code are on an absolute roll, as we covered last week.

Meta to spend more on AI and less on Metaverse

The Wall Street Journal reports on Metas financial planning for 2026:

[Meta] plans to shift spending from the metaverse to artificial-intelligence wearables, according to a person familiar with the matter.

Nothing really surprising here, since the relatively small Metaverse business simply isnt taking off. At the same time, developing VR headsets like Oculus helped Meta to build a hardware engineering function and GPU infrastructure, which are both useful for AI. GPUs purchased for Metaverse purposes can easily be repurposed for AI training, and hardware engineers can work on wearable glasses and other AI gadgets.

Mark Zuckerberg seems hell-bent on not missing out on being the leader in the next big hardware platform category. In 2020, there was a likelihood that VR and AR glasses would be the next big hardware boom. Now, theres some chance it will be wearable AR glasses, or other small gadgets with AI functionality. Meta generates about $60B per year in profit and Zuckerberg has full control over the company, so keeps investing in the Next Big Thing. Is he doing everything possible to avoid the fate of Myspace, which got out-executed by a nimble startup called Facebook, back in the day?

Microsoft failing to sell AI products fast enough

The Information reports:

Multiple Microsoft divisions have lowered how much salespeople are supposed to grow their sales of certain AI products after many of them missed sales-growth goals in the fiscal year that ended in June, according to two salespeople in Microsofts Azure cloud unit. Its rare for Microsoft to lower such quotas for specific products, the people said.

Outside of coding-related use cases, selling AI products seems difficult because AI simply doesnt work that well in other areas.

3. How the engineering team at Oxide uses LLMs

Oxide is a hardware/software startup building a larger server, and the company has a few unique features:

  • Everyone is paid the same base salary

  • Full-remote work is supported, including for hardware engineers (!)

  • Theres an intense writing culture. Most of the interview process is in written form

Now, CTO Bryan Cantrill has published a thoughtful article on how LLMs are used at the company, including some interesting details:

Good use cases:

  • Reading docs: summarizing documents and answering questions about large documents or datasets

  • Editing/polishing docs: after the creative work is completed for a document, the LLM can help polish the last part of the document/writing

  • Lightweight research: instead of using a search engine, an LLM can do better, especially with Googles search quality in decline.

Mixed use cases proceed with caution:

  • Code review: a hit-or-miss some good suggestions with plenty of nonsensical ones.

  • Debugging: can be surprisingly helpful! For tough problems, Bryan says that using LLMs offers little to lose, but perhaps also little to gain. He also warns about debugging fewer hard problems with colleagues: LLMs are no replacement for this, he says.

  • Coding: amazing for experimental and throwaway code. For production code, more involvement from engineers is needed.

Poor use cases avoid:

  • Writing documents. As Bryan puts it: At best, writing from LLMs is hackneyed and clich-ridden; at worst, it brims with tells that reveal that the prose is automatically generated.

I like how the Oxide team keeps using LLMs and experimenting with where they help most, or dont help at all. As engineers, we want to use the right tool for the job: we should figure out when relying on LLMs makes sense, and when it absolutely does not. Id encourage reading the full article for an unusually grounded take on this prescient topic. I cover more about Oxides unique culture in the deepdive, Startups on hard mode: Oxide.

4. Linux officially supports Rust in the kernel

Big news this week for Linux: Rust is officially supported for kernel development. Until now, C was the only officially-supported language in the kernel. From the Linux mailing list:

The consensus among the assembled developers is that Rust in the kernel is no longer experimental it is now a core part of the kernel and is here to stay. So the experimental tag will be coming off. Congratulations are in order for all of the Rust for Linux team.

In April this year, Greg KH (Linux kernel maintainer and Linux Foundation Fellow) appeared on the Pragmatic Engineer podcast to talk about how Linux is built. Back then, there was a back-and-forth in the Linux kernel group about whether to support Rust, or to keep C as the language the kernel is developed in. From the pod:

Gergely: Do you think Linux [the kernel] might at some point support Rust?

Greg KH: So we have 25,000 lines of Rust in the kernel already. Most of that is just bindings. In the latest release, if the kernel crashes, itll put up a QR code. You can take a picture of it to get the crash dump. That code was written in Rust.

Rust for Linux developers have been working for a long time. A couple years ago they came to us and said: we think were ready to do this. Do you want it? And we said: Lets try this experiment. If youre willing to do the work, who am I to say no?

In Rust, you need to have a binding between the C code and the Rust code. Theres an intermediate layer. The kernel written in C has these very opinionated model ideas on how it handles objects and it does its memory model. Rust has its very opinionated model of how it does it. This meshing is tough.

Turns out there are still some performance issues with Rust code versus C code because we can do some tricks in C that they cant do yet in Rust. (...)

But yes, I think Rust needs to come in because it should be easier to write drivers [in Rust.] We have a lot of issues with lifetime rules of when you yank you a device. Devices are dynamic and dealing with these reference counting of things is very tricky to get right [in C.]

He also discussed the benefits preparing for Rust support brings, and some existing resistance:

Rust is forcing us to actually document our C code better and its clearing it up. We can do things better and safer in the C code in order to make Rust easier.

To be fair, a lot of kernel people are very resistant to [the idea of Rust.] They dont like change, dont like different languages. One core kernel developer said I dont like working with a project that has multiple languages just because its tricky.

I see a lot more simple drivers for hardware devices being written in Rust because all they want to do is read and write to some random memory. Its really really easy to do in Rust and you can do it with less code than with C.

Government mandates like the US National Security Agencys (NSA) guidance in 2022 that products be written in memory-safe languages, is also a factor. C is not memory safe, while Rust is, said Greg:

I think weve hit the tipping point where youll start seeing new stuff in [Rust] and we need to do that [use Rust more.] I mean theres mandates from governments that you cant use memory unsafe languages like C in products. And if I want to see Linux succeed, were going to have to change.

I can say: going forward, if you want to write in Rust: write in Rust.

Adding support for a new language is a lot more work than Id assumed; in the case of Rust, the Rust for Linux team has worked tirelessly since 2020 to make this happen. Its nice to see Linux evolving and you can listen/watch the podcast section about Rust support for more information about the pros and cons of developing the Linux kernel in Rust versus C.
